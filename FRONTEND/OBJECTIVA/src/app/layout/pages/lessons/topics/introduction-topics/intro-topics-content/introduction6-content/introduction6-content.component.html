Identifying objects from requirements

What it means?

When you are designing a system or application, the first step is to identify the things or objects that your program will need to handle.

How to do it?

Look at the requirements or the problem you are trying to solve, and ask yourself: “What are the main things involved?”

For example, if you are building a system for a library, the objects are Book, Member, Staff, etc.

Determining object attributes

What it means?

Each object has attributes also called properties or fields, which describe the object’s characteristics or data.

    • Understanding object relationships

    What it means?

    Objects often need to interact with one another. These relationships show how objects are connected or how they work together.

    How to do it?

    Look at how your objects are related. There are a few types of relationships in Object-Oriented Programming, and these are:

    Association

    When objects work together but do not own each other like Teacher and Student who are associated with each other

    Aggregation

    When one object is made up of other objects, but those objects can exist independently like Library has Books, but Books can exist without Library.

    Composition

    A stronger version of aggregation where one object depends on the lifetime of another like Car has an Engine, but without Engine, a Car can’t exist.

    Inheritance

    When one object (child) inherits properties and behaviors from another object (parent) like Dog can be a type of Animal or Toyota Corolla can be type of Car.

    Polymorphism

    When an object can take on different forms or behaviors based on the context like Shape object could represents a Cricle, Rectangle, or Square.
